Purpose: OTEL-02 (inbound server spans + HTTP metrics), OTEL-04 (log correlation) — completes the observability story so a developer can trace a full request in Jaeger and correlate logs.
Output: Instrumented rest.py, 3 test files covering inbound spans, log correlation, and end-to-end integration.

**1. `rest.py` — add FastAPI auto-instrumentation with import guard:**

After the `app = FastAPI(...)` line in `rest.py`, add FastAPI OTEL instrumentation with a graceful import guard:

```python
# OTEL: auto-instrument FastAPI if opentelemetry is available
try:
    from opentelemetry.instrumentation.fastapi import FastAPIInstrumentor
    FastAPIInstrumentor.instrument_app(app)
except ImportError:
    pass  # OTEL not installed — no-op
```

This is the simplest approach: `instrument_app(app)` patches the specific app instance to emit server spans for every inbound request. It automatically creates spans with HTTP semantic conventions (`http.method`, `http.route`, `http.status_code`, `http.server.duration`, `http.server.active_requests`).

**Important:** The `instrument_app(app)` call happens at module import time, which is fine — it only sets up middleware. The actual span creation happens at request time and requires a TracerProvider to be configured (which `init_otel()` does).

**2. `serve.py` — call `init_otel()` at startup:**

Find the serve entrypoint (likely `src/graphql_mcp/adapters/inbound/serve.py` or the `graphql-mcp serve` CLI command). Read the file first. Add `init_otel()` call at the top of the serve function, before `uvicorn.run(...)`:

```python
from graphql_mcp.adapters.outbound.otel_bootstrap import init_otel

def serve(...):
    init_otel()  # Configure OTEL from env vars (no-op if packages absent)
    ...
```

If there's no `serve.py` but the serve command is in `cli.py`, add the `init_otel()` call there instead (in the `serve` Click command, before starting uvicorn). Read `cli.py` to determine the correct location.

**1. tests/test_otel_log_correlation.py — log correlation tests (OTEL-04):**

The `LoggingInstrumentor().instrument(set_logging_format=False)` call from Plan 01's `otel_bootstrap.py` patches Python's `logging.LogRecord` factory to inject `otelTraceID`, `otelSpanID`, `otelTraceFlagged`, and `otelServiceName` attributes into every log record. The `set_logging_format=False` means we don't change the formatter — the attributes are on the record object, available to any handler/formatter.

Create `tests/test_otel_log_correlation.py`:

- `test_log_record_contains_trace_id_and_span_id(otel_setup)`:
  - Set up the LoggingInstrumentor: `from opentelemetry.instrumentation.logging import LoggingInstrumentor; LoggingInstrumentor().instrument(set_logging_format=True)` (use set_logging_format=True in the test so we can see it, or just check record attributes).
  - Actually, the `otel_setup` fixture from Plan 01 should already have LoggingInstrumentor configured via `init_otel()`. If not, add it to the fixture.
  - Better approach: Explicitly instrument logging in this test and capture log records.
  - Create a custom logging handler that captures LogRecord objects:
    ```python
    import logging
    
    class RecordCapture(logging.Handler):
        def __init__(self):
            super().__init__()
            self.records: list[logging.LogRecord] = []
        def emit(self, record):
            self.records.append(record)
    ```
  - Get a logger: `log = logging.getLogger("graphql_mcp.adapters.outbound.http_transport")`
  - Attach the RecordCapture handler.
  - Ensure LoggingInstrumentor is active (it should be from otel_setup, or instrument it explicitly).
  - Create a span context: `tracer = otel_setup["tracer_provider"].get_tracer("test"); with tracer.start_as_current_span("test-op"):` — inside the span, trigger a log emission.
  - The simplest way: within the active span, make a transport call that triggers `logger.warning(...)` (e.g., mock a transport error). OR just call `log.info("test message")` directly.
  - After the log call, check `capture.records[-1]`:
    - `assert hasattr(record, 'otelTraceID')`
    - `assert record.otelTraceID != '0' * 32` (not the zero trace ID)
    - `assert record.otelTraceID == format(span.get_span_context().trace_id, '032x')`
    - `assert hasattr(record, 'otelSpanID')`
    - `assert record.otelSpanID != '0' * 16`
  - Clean up: remove handler, uninstrument.

- `test_log_record_outside_span_has_zero_ids()`:
  - Same RecordCapture setup, but emit a log WITHOUT an active span.
  - Assert `record.otelTraceID == '0' * 32` (zero trace ID when no span).
  - This confirms the instrumentation is working but not injecting fake IDs.

- `test_transport_log_has_trace_context(otel_setup)`:
  - Create an `HttpTransport` pointing at a respx-mocked endpoint that returns 500.
  - Attach RecordCapture to `logging.getLogger("graphql_mcp.adapters.outbound.http_transport")`.
  - Start a span: `with tracer.start_as_current_span("test-query"):` and call `transport.execute("{ hello }")`.
  - The transport will log `"Transport error: ..."` or process the response.
  - Actually, a 500 response doesn't log — it returns a QueryResult with TRANSPORT error. To trigger the `logger.warning("Transport error: %s", exc)` path, mock a connection error instead: `respx.post(...).mock(side_effect=httpx.ConnectError("mock"))`.
  - Assert the captured log record has `otelTraceID` matching the active span's trace ID.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Inbound HTTP → FastAPI middleware | OTEL middleware processes all inbound requests including potentially malicious ones |
| Log records → external collector | Log records with trace IDs exported to external systems |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-04 | Information Disclosure | FastAPI span attributes | accept | Server spans contain route, method, status — standard HTTP observability data, no query bodies or auth tokens. Same as any web framework instrumentation. |
| T-09-05 | Denial of Service | LoggingInstrumentor overhead | accept | LoggingInstrumentor adds ~4 attributes per log record — negligible overhead. No new log volume created, only enrichment of existing logs. |
| T-09-06 | Spoofing | Inbound traceparent header | accept | FastAPI instrumentation accepts incoming W3C traceparent headers for distributed trace context propagation. This is standard behavior — a malicious caller could inject arbitrary trace IDs, but this only affects trace linkage, not security decisions. |
</threat_model>
