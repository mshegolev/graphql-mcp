```python
class QueryDepthError(Exception):
    """Raised when query exceeds GRAPHQL_MAX_QUERY_DEPTH."""
    def __init__(self, depth: int, max_depth: int) -> None:
        super().__init__(f"Query depth {depth} exceeds maximum allowed depth {max_depth}")

        self.max_depth = max_depth
```

**query_guard.py** — Add a `check_query_depth(query_str: str, max_depth: int) -> None` function:

1. Import `from graphql.language.ast import (DocumentNode, FieldNode, FragmentDefinitionNode, FragmentSpreadNode, InlineFragmentNode, OperationDefinitionNode, SelectionSetNode)`.
2. Parse with `gql_parse(query_str)`. On parse error, return silently (let server validate).
3. Build a dict of fragment definitions: `{frag.name.value: frag for frag in doc.definitions if isinstance(frag, FragmentDefinitionNode)}`.
4. Implement a recursive `_measure_depth(selection_set: SelectionSetNode | None, fragments: dict, visited: set) -> int` helper:
   - If `selection_set` is None, return 0.
   - For each selection in `selection_set.selections`:
     - `FieldNode`: `depth = 1 + _measure_depth(selection.selection_set, fragments, visited)`
     - `InlineFragmentNode`: `depth = _measure_depth(selection.selection_set, fragments, visited)` (doesn't add depth — it's a type condition, not a field)
     - `FragmentSpreadNode`: look up fragment name in `fragments` dict. If in `visited` set, skip (cycle protection). Otherwise add to `visited`, measure depth of fragment's `selection_set`, remove from `visited`.
   - Return `max(depths)` or 0 if empty.
5. Measure depth of each `OperationDefinitionNode` in doc. If any exceeds `max_depth`, raise `QueryDepthError(depth, max_depth)`.

Import `QueryDepthError` from `graphql_mcp.domain.errors`.

**Rate Limiter** — Create a `RateLimitMiddleware` class (Starlette `BaseHTTPMiddleware`) at the top of rest.py (after imports):

```python
import hashlib
import time as _time
```

The middleware:
1. Parse `rate_limit` string (e.g., `"100/minute"`) into `(max_requests, window_seconds)` where second=1, minute=60, hour=3600. Parse once in `__init__` from `GraphQLConfig()`.
2. Extract client IP from `request.client.host` (fall back to `"unknown"` if None).
3. Use a sliding-window dict: `_windows: dict[str, list[float]]` keyed by IP. On each request, prune timestamps older than `window_seconds` from the list, then check len.
4. If over limit, return `JSONResponse(status_code=429, content={"error": "rate limit exceeded"}, headers={"Retry-After": str(remaining_seconds)})` where `remaining_seconds = ceil(window_seconds - (now - oldest_timestamp_in_window))`.
5. Otherwise, append current timestamp and call `await call_next(request)`.
6. Only apply to non-health endpoints (skip `/health`, `/ready`).

Add the middleware to the app: `app.add_middleware(RateLimitMiddleware)` — place this AFTER the `app = FastAPI(...)` line but BEFORE route definitions. Note: Starlette middleware executes in reverse add order, so the last-added middleware runs first. Add rate limiter first so it's outermost.

**Header Forwarding** — In each POST endpoint (`graphql_query`, `graphql_raw`, `graphql_entities`), add `request: Request` parameter. Extract `Authorization`, `X-User-Id`, `X-Roles` headers from `request.headers`. Pass them to the transport by modifying the approach:

Create a helper function `_extract_forwarded_headers(request: Request) -> dict[str, str]`:
```python
_FORWARDED_HEADERS = ("authorization", "x-user-id", "x-roles")

def _extract_forwarded_headers(request: Request) -> dict[str, str]:
    return {k: v for k, v in request.headers.items() if k.lower() in _FORWARDED_HEADERS}
```

For header forwarding to actually reach upstream, the `HttpTransport._client` needs per-request headers. The cleanest approach: modify the POST endpoints to inject forwarded headers into the transport's httpx client before calling. Since `httpx.Client.post()` accepts a `headers` param that merges with client-level headers, update `HttpTransport.execute()` and `post_raw()` to accept an optional `extra_headers` parameter. BUT — to avoid modifying the domain-facing transport interface in this plan (transport changes are Plan 02's scope), use a simpler approach:

In `_get_client()` and in the endpoint handlers, set request-scoped headers on the transport's httpx client: `client._transport._client.headers.update(forwarded_headers)`. This works because FastAPI runs synchronously in a thread pool by default for sync endpoints. After the response, these persist (acceptable — they'll be overwritten on next request).

Actually, the cleaner pattern: add `extra_headers` kwarg to `HttpTransport.execute()` and `HttpTransport.post_raw()` that gets passed as `headers=` to `self._client.post()`. httpx merges per-request headers with client-level headers. This is a small, additive change to http_transport.py:

In `http_transport.py`, update `execute()` and `post_raw()`:
```python
def post_raw(self, body: dict[str, Any], extra_headers: dict[str, str] | None = None) -> QueryResult:
    try:
        response = self._client.post("", content=self._codec.encode(body), headers=extra_headers)
    ...

def execute(self, query: str, variables: dict[str, Any] | None = None, extra_headers: dict[str, str] | None = None) -> QueryResult:
    body: dict[str, Any] = {"query": query}
    if variables:
        body["variables"] = variables
    return self.post_raw(body, extra_headers=extra_headers)
```

Similarly for `async_http_transport.py`. These are backward-compatible (default None). Then in rest.py endpoints, pass `extra_headers=forwarded` to `client.query()` etc. BUT — `GraphQLClient.query()` doesn't accept extra_headers. The simplest correct approach:

Modify `GraphQLClient.query()`, `raw()`, `entities()` in `lib.py` to accept and forward `extra_headers: dict[str, str] | None = None`. The transport's execute/post_raw get the same kwarg. This is purely additive (default None preserves backward compat for all existing callers).

```python
def graphql_query(req: QueryRequest, request: Request) -> dict[str, Any]:

    client = _get_client()
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → REST API | Untrusted input: query strings, headers, IP addresses |
| REST API → upstream GraphQL | Semi-trusted: forwarding validated requests with auth context |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | D (Denial of Service) | REST /graphql/query | mitigate | Query depth limit via AST analysis in check_query_depth() — reject before upstream |
| T-10-02 | D (Denial of Service) | REST all endpoints | mitigate | IP-based sliding-window rate limiter in RateLimitMiddleware — 429 + Retry-After |
| T-10-03 | S (Spoofing) | Header forwarding | mitigate | Only forward explicitly allowed headers (Authorization, X-User-Id, X-Roles) — no wildcard forwarding |
| T-10-04 | T (Tampering) | Rate limiter state | accept | In-memory state — resets on restart. Acceptable for single-instance; Redis upgrade path for HA |
| T-10-05 | E (Elevation) | Header injection | mitigate | Whitelist approach: only 3 named headers forwarded, not arbitrary X-* headers |
</threat_model>
